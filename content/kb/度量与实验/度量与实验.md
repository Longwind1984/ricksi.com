# M2. AI 产品度量与实验

> **回答的核心问题**：这个 AI 功能上线后，怎么知道它是好是坏？怎么持续改进？
>
> **在 v2 框架里的位置**：`0框架.md` 的 M0技术 / M1产品架构 / **M2度量实验** / M3商业 / M4竞品 / M5风险 + OS 底座里，M2 是"上线后怎么度量"这一块——v1 的最大盲区（技术、工程、产品设计都有家，唯独度量散落各处）。本页是 M2 的单篇 synthesis。

## 核心主张（Synthesis）

> **AI 产品的度量不是"传统指标 + 一个准确率"，而是三件事的耦合：非确定性输出的统计化度量、模型指标到业务指标的非线性归因、以及度量数据反向喂回训练的飞轮。** 把任何一件当独立任务做都会失败：只看模型指标会撞 [Goodhart](/kb/基础知识库/goodhart-定律/)；只看业务指标会归因失真；只做度量不做飞轮会浪费掉 AI 产品独有的、传统产品没有的资产——**使用数据本身就是训练数据**。

三件耦合里，第三件是 AI 产品相对传统软件的**结构性差异**，也是 [p306](/kb/产品设计与交互范式/p306-数据飞轮与反馈回路设计/) 的命题源头。前两件是 [c14](/kb/基础知识库/c14-模型评估体系与-goodhart-陷阱/) 在产品侧的延伸——c14 讲"怎么测模型"，本模块讲"怎么知道功能好不好"，两者相邻但不同。

## 2.1 三层指标体系

传统互联网产品的指标是两层（行为指标 → 业务指标）。AI 产品必须三层，且层间关系非线性：

```
模型性能层（技术侧）   →   产品体验层（用户侧）   →   商业结果层（业务侧）
   看不见，但决定上限        用户直接感知           老板/投资人看
```

> 注：本页正文用"模型性能 / 产品体验 / 商业结果"三层做归因主轴。落地填表时还可在三层之外单列一个 **数据回流层**（哪些用户行为能变成下一轮训练燃料）——它不参与"模型→业务"的归因链，但参与飞轮，见 §2.4 与文末模板。

### 2.1.1 模型性能层（技术侧）

| 指标 | 定义 | 产品含义 | 陷阱 |
|---|---|---|---|
| 任务完成率 Task Completion | 用户的请求是否被正确执行 | 最接近"功能正不正常"的代理 | 需定义"完成"——是模型输出对，还是用户拿到想要的东西 |
| 幻觉率 → 人工接管率 | 模型编造事实的频率，落地为"人工接管率" | 直接映射客服/审核人力成本 | 见 [幻觉](/kb/基础知识库/幻觉/)，幻觉不可消除，只能压到阈值下 |
| TTFT（首字延迟） | 用户等到第一个 token 的时间 | 流式体验的"快不快"感知锚 | 见 c05，受 Prefill 阶段约束 |
| TPOT（每 token 生成时间） | 内容流出的速度 | "卡不卡"感知 | output token 受显存带宽瓶颈（自回归） |
| 检索命中率 / MRR | RAG 场景：召回准不准、排序对不对 | 决定第一个答案是不是最相关 | RAG 体验生死线，常被工程团队低估（见 m203） |

### 2.1.2 产品体验层（用户侧，AI 特有）

这一层是传统 PM 经验能迁移、但要叠加概率系统修正的部分：

| 指标 | 定义 | 为什么 AI 特有 |
|---|---|---|
| **采纳率** Acceptance Rate | 用户是否使用了 AI 生成的结果 | 确定性系统里"功能用了=功能对了"；概率系统里"用了"≠"对了" |
| **编辑距离** Edit Distance | 用户在 AI 输出基础上改了多少 | 距离越小越好——是"AI 接近用户意图"的连续度量，比点赞/点踩信息量大得多 |
| **重试率** Retry Rate | 触发"重新生成"的频率 | 每次重试 = 双倍成本（见 m209 §2.6.1）+ 满意度负信号 |
| **会话深度** Session Depth | 用户交互了几轮才完成任务 | 深度过深 = 单轮解决率低；但 Agent 任务天然多轮，需区分场景 |
| **逃逸率** Escape Rate | 用户在 AI 功能里放弃，转手动完成 | 最强的负面信号——比"不满意"更严重，是"不信任" |

> **PM 直觉**：这五个里，**编辑距离和逃逸率是信息密度最高的两个**。点赞/点踩是显式反馈（稀疏、有偏），编辑距离是隐式反馈（密集、无偏）——这直接接 [p306](/kb/产品设计与交互范式/p306-数据飞轮与反馈回路设计/) 的数据采集哲学。

### 2.1.3 商业结果层（业务侧）

> 本层只点出"AI 特有之处"，LTV/COGS 的完整商业模型在 0434 与 [m209](/kb/工程化与落地架构/m209-推理成本控制手册/)，此处不展开。

| 指标 | AI 特有之处 |
|---|---|
| 效率提升量化 | 用 AI 前后完成同一任务的时间对比——但需控制"用户学习曲线"混淆变量 |
| 单次交互 COGS | 算力成本随使用量线性增长（见 m209），与传统软件"边际成本趋零"根本不同 |
| LTV / CAC | AI 产品的 LTV 测算必须把"单用户算力消耗"纳入 COGS，否则毛利算错（商业模型详见 0434） |
| 留存与付费转化 | AI 功能的留存曲线形状与传统产品不同——常有"惊艳期高、工具期回落"的衰减 |

## 2.2 指标间的因果链与阈值效应

> **核心张力**：不是所有模型性能提升都会转化为业务指标改善。

### 2.2.1 阈值效应（threshold effect）

某些场景下，准确率必须跨过一个门槛才会引发用户行为变化：

- 客服场景：幻觉率从 5% → 3% 可能业务指标纹丝不动；从 3% → 1% 可能带来留存跃升——因为 1% 是用户"不再每次核查"的心理阈值。（**示意算例**，非某产品实测数据）
- 编程 Agent：单步成功率 95% → 97% 是质变，90% → 92% 用户无感。原因是**复合错误的数学**——多步任务里单步成功率被指数放大：20 步任务下 0.95²⁰ ≈ 36%、0.97²⁰ ≈ 54%（**这是数学推导出的示意算例，用来说明放大效应，不是任何产品的真实指标**；复合错误机制见 [c10 §10.3](/kb/基础知识库/c10-agent-技术栈与工具调用/)）。

**PM 含义**：上线前的 ROI 测算不能假设"准确率提升 X% → 业务指标提升 Y%"是线性的。要找到自己产品的**那个阈值**，把资源投在跨过它，而不是均匀地"再准一点"。

### 2.2.2 感知质量 vs 实际质量

这和"感知安全 ≠ 实际安全"是同构问题（Rick 在滴滴安全产品上的经验，展开见 0423 自我民族志，此处只借作类比）：

- AI 实际幻觉率低，但**输出自信、无引用** → 用户感知不可靠 → 逃逸率高。
- AI 实际幻觉率高，但**带引用、带置信度** → 用户感知可靠 → 采纳率高。

**PM 含义**：[信任架构](/kb/产品设计与交互范式/p305-信任架构与可解释性设计/) 不是附加题，是度量体系的一部分——感知质量独立于实际质量，需单独度量。

### 2.2.3 三层归因链条

```
模型指标（准确率↑）→ 产品指标（采纳率↑？）→ 业务指标（留存↑？）
        ↑                    ↑                    ↑
   每一跳都可能断           阈值效应            感知质量干扰
```

建立这条链是 AI PM 度量工作的核心交付物——**没有这条链，所有"模型升级了所以业务变好了"的归因都是故事**。

## 2.3 AI 产品的 A/B 实验设计

> 传统 A/B 的两个隐含假设在 AI 产品里**不成立**：① 确定性输出（同输入同输出）；② 独立样本（用户间不互相影响）。

### 2.3.1 三个失效点

1. **非确定性**：同一用户同一输入，模型可能给不同输出。传统"对照组看到 A、实验组看到 B"的干净对照被污染——你看到的差异里有模型随机性的噪声。
2. **多变量耦合**：一次升级可能同时改了 Prompt、模型版本、检索策略。哪个变量贡献了提升？传统 A/B 回答不了。
3. **长尾集中**：AI 产品的问题往往集中在少数极端场景。均值指标（如平均满意度）可能掩盖 5% 用户的灾难性体验。

### 2.3.2 可落地的方法

| 问题 | 方法 |
|---|---|
| 非确定性噪声 | 同一用户多次请求取统计量（均值/分位），而非单次；或加大样本量吃掉方差 |
| 多变量耦合 | **分层实验 / 正交实验**（层内只改一个变量）；或先离线跑消融实验（ablation）再上线 |
| 长尾集中 | 除均值外，必看 P90/P99 分位 + **bad-case 专项评估**（见 0412 评测专题 的 bad-case 库思路） |
| "好"是场景依赖 | 同一输出，创意场景是优点（发散）、合规场景是灾难（不精确）——实验指标必须按场景分层，不能合在一起看 |

### 2.3.3 模型版本切换的实验范式

这是 AI PM 最高频的实验场景——**跨大版本切换模型时**（例如从一代旗舰模型升到下一代，或在不同厂商旗舰间迁移；具体型号名与发布时间〔待核实〕，不在本页写死）。关键问题：

- **回归测试**：新模型在旧场景上退化了没？这是比"新场景提升多少"更优先的问题——因为老场景的用户基数大。建议维护一个**回归测试集**（见 [c14](/kb/基础知识库/c14-模型评估体系与-goodhart-陷阱/) 的评估体系，落地模板见文末 §模板-3）。
- **Prompt 是否要重调**：换模型后旧 Prompt 可能失效。需在实验里隔离"模型效应"和"Prompt 效应"。

## 2.4 数据飞轮的工程化

> 这是 AI 产品相对传统软件的**结构性优势**，也是 [p306](/kb/产品设计与交互范式/p306-数据飞轮与反馈回路设计/) 的命题。本节补 p306 没展开的"工程管道"层。OS 模块里"PRD 数据采集段该写什么"会回链本节，不重画此管道。

### 2.4.1 三种数据捕获哲学对比

| 产品 | 捕获机制 | 数据哲学 |
|---|---|---|
| [ChatGPT](/kb/ai-公司与产品/chatgpt/) | 点赞 / 点踩 / 重新生成 | **显式反馈**——稀疏、有偏（只有强烈情绪的用户才点），但信号强 |
| Copilot | Tab 采纳 / Esc 拒绝 | **隐式行为**——密集、无偏（每个补全都被记录），信息量大 |
| Midjourney | V 变体选择 / 放大 | **选择即数据**——选择本身揭示了用户在"哪个方向"上满意，比二值反馈丰富 |

**PM 决策**：选哪种取决于产品形态。对话框产品只能做显式；行内补全产品能做隐式；创意工具能做选择式。**隐式 > 显式**——能在交互里隐式采集的，绝不弹窗问用户。

### 2.4.2 反馈数据 → 训练数据的转化管道

```
前端交互（采纳/编辑/停留/复制）
   ↓ 采集
原始行为日志
   ↓ 清洗（去噪、去隐私、去 PII）
结构化反馈对（input, output, 偏好信号）
   ↓ 标注（人工校验 + LLM-as-Judge 辅助）
SFT / DPO 训练样本
   ↓ 训练
迭代后的模型
   ↓ 上线
新的交互……（飞轮闭环）
```

**PM 在这条管道里的职责**：定义"偏好信号怎么从交互里提取"。这不是工程问题，是**产品设计问题**——采集点埋得对不对，决定了飞轮转不转得起来。落地时把行为信号拆成四类：显式正负反馈 / 隐式编辑反馈 / 引用与证据反馈 / 人工兜底反馈（对应 §模板-4 的回流记录卡）。

### 2.4.3 冷启动期的数据策略

没有用户反馈数据时怎么办（新产品 / 新功能）：

1. **合成数据**（synthetic data）：用强模型生成初始训练样本。风险：合成数据的偏差会被放大，形成回音壁。
2. **人工标注 + 内部 dogfooding**：团队自己用，标注一致性 > 标注量（见 0412 的 IAA 计算）。
3. **借用公开数据集**：用公开 benchmark 做基线，但注意数据污染（见 0412 的 Benchmark 污染）。

## 2.5 评估体系搭建实操

> 这节和 0412 评测系统化专题 有重叠，但视角不同：0412 讲"怎么测模型/系统"（研究侧），本节讲"PM 怎么搭一套能用的评估"（落地侧）。深层方法（LLM-as-Judge 偏见机制、IAA 计算）只点名 + 回链，不重写。

### 2.5.1 离线评估 vs 在线评估的分工

| 类型 | 作用 | 局限 |
|---|---|---|
| 离线评估 | 上线前把关，回归测试，不污染用户 | 和真实分布有 gap（见 0412 的 benchmark 失真） |
| 在线评估 | 上线后真实表现，A/B | 有用户体验风险，跑得慢 |

**铁律**：离线过了才上线；但离线过了**不等于**在线会好——必须两层都跑。落地用 §模板-2 的"离线-在线协同表"对账，四象限处置见下。

**离线-在线四象限处置**：

| 区域 | 含义 | 处理动作 |
|---|---|---|
| 离线涨，线上涨 | 真正有效 | 扩量、沉淀经验 |
| 离线涨，线上不涨 | 指标未穿透体验 | 查交互、流量结构、使用门槛 |
| 离线不涨，线上涨 | 可能改善了感知层而非能力层 | 拆解 UI、默认入口、审阅成本 |
| 离线跌，线上涨 | 高风险假繁荣 | 重点抽查长尾失败与合规风险 |

### 2.5.2 LLM-as-Judge 的搭建与已知偏见

用强模型给弱模型输出打分（偏见机制的完整讨论见 0412 的 LLM-as-Judge，本页只列 PM 需知的结论）。已知偏见：

- **位置偏见**：倾向于选第一个/最后一个。
- **冗长偏见**：倾向于更长的回答。
- **自我偏好**：某些 judge 倾向于偏好同源模型的输出。

**PM 含义**：LLM-as-Judge 不是免维护的。需定期用人工校准 judge 本身（judge 的 judge）。

### 2.5.3 人工评估的标注指南设计

- **标注一致性 > 标注量**：10 个标注者各标 100 条不如 3 个标注者交叉标 300 条算 IAA（见 0412 的 Cohen Kappa）。
- 标注指南要给**反例**而非只给正例——告诉标注者"什么不算好"，比"什么算好"更能拉齐标准。

## 反例与张力

- **反对意见**："这三层指标太重了，小团队跑不动。" 反驳：不是每个产品都要全跑三层，但**心智模型必须是三层**——哪怕只度量一层，也要知道自己在哪一层、漏了哪两层。不知道自己漏了什么，比漏了什么更危险。
- **张力**：数据飞轮听起来是免费午餐，但飞轮转起来有前提——用户的隐式反馈确实能反映"真实偏好"。创意类产品（用户自己也不知道想要什么）的隐式反馈噪声大，飞轮可能空转。这是 Midjourney 选择"V 变体"而非"采纳/拒绝"的原因——它在帮用户把模糊偏好显式化。
- **指标过载 vs 指标空壳**：把这一层做成"数据分析百科全书"会过宽、失去 AI PM 决策焦点；只保留模板而不解释指标间关系，又会退化成运营表格包。三层框架 + 文末模板正是为了卡在两者之间。

### §模板-4 数据回流记录卡

把前端行为变成可再利用样本。关键不是"有没有反馈按钮"，而是能否把行为信号拆类并判断能否入训练集。

```markdown
## 数据回流记录

- **事件类型**：采纳 / 拒绝 / 编辑 / 重试 / 引用点击 / 人工接管
- **原始上下文 ID**：
- **建议标签**：hallucination / irrelevant / over-refusal / latency / formatting / unsafe
- **是否可入训练集**：是 / 否
- **若否，原因**：隐私 / 噪声 / 标注不足 / 样本冲突
- **建议去向**：SFT / preference data / eval set / red-team set / rule patch
```
